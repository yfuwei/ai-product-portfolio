本地个人知识库 RAG 系统

一个面向个人知识管理场景的本地 AI 检索系统。它通过本地 embedding、向量数据库、来源引用和轻量 GUI，把 Obsidian 笔记与本地文档转化为可检索、可追溯、可迁移的个人知识层。

Overview

不是聊天机器人，而是可信的个人知识检索工作台

项目核心不是让模型“会说”，而是让用户在大量个人笔记和本地资料中找得到来源、看得到依据、能回到原文复核。

已索引文件163

162 个 Obsidian Markdown 文件 + 脱敏文档测试样例。

向量片段390

385 个 Obsidian chunks + 5 个文档 chunks。

运行方式本地

Ollama embedding、Chroma 向量库、FastAPI GUI。

机器约束Intel i5

按 2020 款 MacBook Pro 16GB 内存设计轻量方案。

Problem Definition

问题定义

个人知识库越大，问题越不是“没有资料”，而是资料找不回、答案不可信、隐私边界不清楚。

痛点具体影响产品解法
关键词搜索弱用户记得大概意思，但不记得文件名和关键词。用本地 embedding 做语义召回。
LLM 容易幻觉直接问模型会得到流畅但无依据的回答。先检索来源，再基于来源生成答案。
私人资料敏感求职资料、项目复盘、私人笔记不适合上传云端。embedding、向量库、缓存和 GUI 都在本地运行。
命令行门槛高终端可以调试，但不适合日常使用。提供 Web GUI，覆盖搜索、问答、索引和状态。

Target Users & Use Cases

目标用户与核心场景

当前版本首先服务个人知识工作流，再以作品集形式展示完整的产品设计与系统实现能力。

知识工作者

有大量笔记、资料和项目沉淀，需要快速找回历史资料和决策依据。

求职转型者

需要整理作品集、项目经历和面试材料，从分散笔记中提取可讲述内容。

AI 产品学习者

希望理解 RAG 如何接入真实个人工作流，而不是停留在脚本 demo。

场景用户问题产品价值
找回历史规则我之前写过的 AI 协作规则在哪里？用语义检索找回相关笔记。
准备项目讲述本地 RAG 项目如何介绍？聚合项目简介、压缩版和验证资料。

换电脑迁移换电脑时需要迁移哪些内容？明确代码、模型、向量库、缓存和路径配置。

Product Scope

产品目标与非目标

明确边界能避免把本地 RAG 包装成过大的平台，也能让项目更可信。

产品目标

支持 Obsidian Markdown 和本地文本类文档索引；支持自然语言语义检索；支持基于来源的回答生成；回答必须展示引用来源；支持低配置电脑本地运行；提供 GUI，降低日常使用门槛。

非目标

不做企业级多用户权限系统；不默认上传私人资料到云端；不追求大模型复杂推理；不把生成答案当成最终事实；当前阶段不完整覆盖扫描 PDF、图片和音视频。

System Architecture

系统架构

架构拆成数据源、处理、向量、检索、生成和体验六层，避免把 RAG 简化成“接一个聊天框”。

系统架构图

展示从 Obsidian 与本地文档到文本处理、embedding、Chroma、检索、生成和 GUI 的完整链路。
层级职责设计重点
数据源层Obsidian Markdown、本地文档保持原始资料不被修改。
处理层转换、清洗、切分让长文档变成可检索片段。
向量层embedding、Chroma支持语义召回和本地持久化。
检索层query embedding、相似度搜索优先找准来源。
生成层prompt、LLM 回答基于来源回答，不脱离上下文。
体验层Web GUI让非终端使用成为可能。

Data & Indexing

数据接入、切分与索引机制

索引机制的重点是稳定、可追溯、可迁移，而不是一次性追求最大资料量。

数据接入

优先接入 Obsidian Markdown 笔记、本地文本类文档和少量文档转换样例；证件、合同、财务、医疗等敏感档案默认排除。

Chunk 策略

切分目标不是越细越好，而是让片段信息完整、语义清晰、便于阅读，并能回到来源文件定位。

Embedding 与向量库

本地 embedding 保证资料不出本机；Chroma 本地持久化简单，适合 MVP 快速验证和来源元数据保存。

Prompt & Citation

Prompt 与引用机制

引用不是技术附属信息，而是用户信任系统的关键界面元素。

Prompt 原则

只能基于提供的上下文回答；如果上下文不足，要说明不确定；回答后必须列出引用来源；不编造未出现在来源中的事实。

引用元数据

每个 chunk 保留来源文件、标题或章节、片段文本、来源名称和 chunk id。GUI 展示时优先让用户看到依据。

产品原则：先展示来源，再组织答案。个人知识库场景里，正确依据比流畅回答更重要。

Design Decisions

关键设计决策与取舍

项目价值不只是跑通 RAG，而是围绕隐私、可信度、性能和日常使用做出清晰取舍。

决策收益成本
RAG 而非微调动态接入新增资料，并保留来源。需要维护索引与召回质量。
本地部署私人资料不上传，控制权更强。模型能力和速度受本机限制。
引用优先降低幻觉，方便回到原文复核。回答流程比普通聊天更复杂。
GUI 优先日常可用，更像产品而非脚本。需要额外维护前端体验。
rerank 后置MVP 更轻，适配低配置电脑。大规模资料下排序仍有优化空间。

Limitations

限制与风险

完整案例页需要诚实说明边界，这也是 AI 产品可信度的一部分。

小模型能力有限

适合辅助总结，不适合复杂推理。

大批量索引占 CPU

需要小批量、分阶段执行索引任务。

路径变化需要维护

换电脑或移动目录后需要确认数据源配置。

格式覆盖有限

扫描 PDF、图片和复杂表格暂未完整覆盖。

Product Insights

产品洞察

这个项目的重点不是“会搭 RAG”，而是理解 AI 产品如何获得用户信任。

AI 产品的关键不是能回答，而是能被信任

用户不缺一个会说话的模型，缺的是能找回真实资料、解释依据、减少误判的系统。

本地 RAG 的价值在于控制权

云端工具能力强，但不是所有资料都适合上传。本地方案在隐私、可控和长期维护上更适合个人知识库。

GUI 是从 demo 到产品的分界线

同样的 RAG 能力，如果只能在终端使用，更像开发工具；有 GUI 才更接近日常产品。

Documents

可读材料入口

页面用于快速阅读，PDF 用于外发。Markdown 文件仅作为内部维护稿。
