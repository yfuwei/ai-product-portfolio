本地知识助手（RAG → Agentic RAG）

从“语义检索工具”升级为“具备决策能力的 AI 知识助手”。

让用户从“找文件”转变为“直接获取有依据的答案”：V1 用 RAG 解决资料召回与引用溯源，V2 用 Router、Tool Calling 和 Memory 把系统升级为任务执行入口。

RAG Agent Tool Calling Memory 本地部署 Agentic RAG
验证结果 10 条真实任务问题：9 条命中，1 条基本命中，回答可用性约 8/10
项目定位 不是普通 RAG Demo，而是一个带系统演进、评估和边界意识的 Agentic RAG 案例

TL;DR

项目速览

不看全文也能理解 80%：问题、方案、结果和亮点都围绕“从找文件到找答案 + 找依据”。

问题
  • 知识分散，靠关键词找不准。
  • 直接问 LLM 容易无依据。
  • 连续任务缺少上下文。
方案
  • V1：RAG 解决检索与引用。
  • V2：Agent 解决决策与连续使用。
  • 默认本地部署与只读边界。
结果
  • 10 条真实任务问题。
  • 9 条命中，1 条基本命中。
  • 回答可用性约 8/10。
亮点
  • RAG 工具化。
  • Agent 决策层。
  • 可解释执行过程。

Users & Scenarios

目标用户与核心场景

本项目面向需要频繁查找历史资料、整理项目材料和复盘经验的个人知识工作者，而不是泛泛的“普通用户”。

AI 产品经理候选人

需要整理作品集、复盘项目、准备项目问答，并从历史材料中快速找到可引用依据。

项目经理 / 咨询顾问

需要从历史文档中快速找依据、总结项目、沉淀方法，减少重复翻资料和重新组织表达。

研究型知识工作者

需要在大量笔记中进行语义检索、追溯来源和连续整理，让知识库真正服务长期输出。

查资料 找依据 总结 复盘 生成表达 连续追问
查资料 不记得文件名，也能找回相关资料

用户用自然语言描述问题，系统通过语义召回覆盖同义表达和跨文件线索。

做复盘 从多个文档中提取背景、方案和结果

把分散在笔记、项目材料和设计文档里的信息重新组织成可复用结论。

生成表达 整理成项目介绍、项目表达或作品集内容

让 AI 不是凭空写作，而是在已有资料和来源依据上辅助生成表达。

Problem

问题定义

用户的问题不是“如何问 AI”，而是如何让 AI 在本地知识中“找得到 + 信得过 + 用得顺”。

找得到 语义

资料分散在 Obsidian、项目材料和本地文档中，关键词搜索无法覆盖同义表达和跨文件线索。

信得过 来源

LLM 回答必须绑定来源片段，让用户能回到原文复核，而不是只接受模型结论。

控得住 本地

embedding、向量库、缓存和回答链路留在本机，适合私人知识管理场景。

用得顺 上下文

用户会连续追问、对比和整理项目，系统需要判断任务类型并承接最近上下文。

Why Not Existing Tools

为什么不直接用现有方案

这个项目不是为了追热点做 RAG，而是因为现有工具只能解决一部分问题，无法同时满足本地资料接入、来源追溯、持续更新和连续任务。

方案 能解决什么 主要问题 本项目判断
ChatGPT 直接问 通用解释、写作、总结能力强。 不知道本地资料；手动上传不适合长期维护；来源追溯不稳定。 适合作为生成层，不适合作为本地知识入口。
Notion / Notion AI 在线文档组织和团队协作体验好。 资料分散在 Obsidian、本地文件和 OneDrive；隐私和来源粒度不完全匹配。 不作为主方案。
Obsidian / 关键词搜索 适合精确查找文件名和关键词。 难覆盖同义表达、跨文件线索和自然语言问答。 保留为辅助。
本地 RAG 支持本地资料接入、语义检索和来源追溯。 需要维护 chunk、索引和召回质量。 作为 V1 主方案。
Agentic RAG 支持任务判断、工具调用和连续追问。 引入 Router 误判、Memory 污染等新风险。 作为 V2 升级方向。
本项目的判断是：用 RAG 解决“本地知识接入 + 来源追溯”，再用 Agentic RAG 解决“任务判断 + 工具调用 + 连续追问”。

V1

V1：基础 RAG 方案

先解决最核心的问题：让本地资料可检索、回答可追溯、工具可日常使用。

一句话说明

V1 是一个本地 RAG 工作台：完成文档导入、向量检索、带引用回答和 Web GUI。

核心能力
  • 本地 embedding + Chroma 向量检索。
  • 基于召回片段生成回答，并展示引用来源。
  • 支持 Obsidian 笔记和本地文档的统一检索。
  • 提供 GUI，降低命令行使用门槛。

检索工作台

先找资料，再组织答案。

带引用回答

回答和来源绑定，降低黑盒感。

Why Upgrade

V1 为什么不够

V1 已经能回答问题，但还不是一个真正“会判断”的助手。

所有问题都强制检索

无论是查资料、解释概念还是延续上一轮问题，都走同一条链路。

没有任务判断

系统只能被动检索，不能识别用户是在问答、总结、对比还是整理项目。

没有上下文记忆

每轮对话相对独立，连续追问时用户需要反复补背景。

V2

V2：Agentic RAG（Agent 知识助手升级）

从“用户提问，系统回答”升级为“用户提出目标，系统判断路径并调用工具完成任务”。

一句话升级

LLM 不再只是生成答案，而是先判断要不要查资料，再决定如何完成任务。Agent 的引入，本质上是把系统从“信息检索工具”升级为“任务执行入口”。

架构说明

User Query 先进入 Router；需要依据时调用 `search_docs` Tool；再结合 Memory 生成最终答案；Agent Core 负责整条链路调度。

Router

先判断，再决定是否检索

区分 direct 和 retrieve，让流程更贴近真实意图。

Tool

把检索变成标准能力

将 RAG 封装成 `search_docs`，为后续多工具扩展留接口。

Memory

支持连续追问

保留最近上下文，减少重复输入，提升连续任务体验。

Agent Core

负责调度整条链路

连接决策、工具调用、上下文组装和最终回答。

系统架构

从数据源、文本处理、向量层到 Agent 控制层、生成层和体验层。

查询流程

User → Router → Tool(RAG) → Answer → Memory，先判断任务，再调用工具。

1 User

输入问题或任务目标。

2 Router

判断是 direct 还是 retrieve。

3 Tool

需要依据时调用 `search_docs`。

4 Reasoning

结合来源和上下文组织答案。

5 Memory

保存最近对话，支持下一轮继续。

Evaluation

评估体系：V1 验证检索可信，V2 验证 Agent 决策

测试问题来自真实个人知识库使用场景，而非人工构造的标准问答集。评估重点不是回答是否流畅，而是来源是否召回、回答是否基于来源、Agent 是否判断对任务类型。

测试问题 10

覆盖协作规则、GUI 使用、索引维护、迁移和项目介绍。

召回结果 9+1

9 条命中，1 条基本命中，基本命中表示召回方向正确但仍需筛选。

回答可用性 8/10

人工判断是否基于来源、是否能辅助复盘。

Agent 判断 4类

检索、问答、索引维护和澄清类任务。

分层评估

检索层、生成层、Agent 层和边界层同时看，避免只看模型输出是否好看。

产品判断

V2 的重点不是证明“Agent 更高级”，而是验证它是否真的能让系统从固定问答链路升级为可控的任务链路。

因此评估必须同时看：是否找得到、是否能追溯、是否判断对路径、是否把高风险动作挡在确认边界外。

V1：基础 RAG 验证
验证重点 判断方式
是否找得到 正确来源是否进入 Top-K。
是否答得出 回答是否能基于来源组织。
是否能追溯 用户是否能回到原文复核。
是否可维护 文件更新后索引是否能刷新。
  • 10 条真实任务问题。
  • 9 条命中，1 条基本命中。
  • 回答可用性约 8/10。
V2：Agentic RAG 验证
验证重点 判断方式
Router 是否判断正确 direct / retrieve / clarify 是否符合预期。
Tool 是否正确调用 需要本地资料时是否调用 search_docs。
Memory 是否有效 连续追问是否能继承上下文。
Trace 是否可解释 是否展示 route decision、tool call、sources。
Safety 是否可控 写入 / 删除类动作是否要求确认。

Badcase Review

Badcase 复盘：从 Demo 到可迭代产品

这个项目不只展示成功案例，也记录系统在真实使用中可能出现的问题，并将 badcase 转化为后续优化策略。

Badcase 我是怎么发现的 优化动作 产品判断
语义相近导致误召回 问“本地知识助手项目怎么介绍”时，召回了其他 RAG 项目资料。 增加 project_name、source_type、path 等元数据；支持来源筛选。 相似度高不等于业务相关，RAG 需要业务元数据。
长文档总结片面 总结产品设计文档时，只覆盖某个章节。 增加章节摘要索引；总结类问题优先召回目录和章节摘要。 基础 RAG 适合局部问答，长文档总结需要结构化策略。
Router 误判 “我这个项目怎么讲”被误判为 direct，没有检索本地资料。 给 Router 增加本地资料信号规则；低置信时 clarify。 Agent 的第一步是路径判断，Router 错了后面都错。
写入类请求风险 用户要求写回 Obsidian 时，存在误写和覆盖风险。 当前默认只读；写入前必须预览和确认。 Agent 能力边界必须先于能力扩展。
V1 的 badcase 主要发生在检索和生成质量；V2 的 badcase 主要发生在决策、上下文、工具调用和安全边界。

Comparison

对比传统方案与 V1/V2 演进

本质变化是从“找文件”升级为“找答案 + 找依据”，再进一步升级为“会判断路径的任务执行入口”。

维度 传统方式 本系统
找资料 关键词搜索、按目录翻文件。 自然语言语义检索，覆盖同义表达和跨文件线索。
找答案 打开多个文件后手动拼结论。 先召回来源，再组织可复核答案。
可信度 依赖用户记忆和人工判断。 回答绑定来源片段，可回到原文确认。
连续任务 每次重新搜索、重新交代背景。 Memory 保留最近上下文，Router 判断是否继续检索。
维度 V1：基础 RAG V2：Agentic RAG
架构 固定的 Retrieve → Generate 链路。 Query → Router → Tool / Direct → Memory → Answer。
决策能力 默认每次检索，没有任务判断。 先判断是否检索，再决定调用哪条路径。
用户体验 更像一个带引用的知识检索工具。 更像一个能理解上下文、会判断的知识助手。

Product Decisions

关键产品决策与 Trade-off

这个项目的重点不是堆功能，而是在不同方案之间做取舍：哪些能力先做，哪些能力暂不做，哪些风险必须控制。

决策 为什么这样做 放弃 / 代价
先做基础 RAG，而不是一开始做 Agent 先验证“找得到、答得出、能追溯”的基础闭环。 V1 不够智能，无法判断任务路径。
RAG 封装为 search_docs Tool 让检索能力从固定流程变成 Agent 可调用能力。 需要额外定义工具输入输出和失败状态。
Memory 只保留短期上下文 支持连续追问，同时降低上下文污染风险。 不做长期用户记忆。
Agent 默认只读，不自动写回 Obsidian 避免误写、覆盖和污染本地资料。 自动化程度降低。
当前不优先 fine-tuning 项目核心是知识接入、更新和来源追溯，不是风格训练。 Router / 格式稳定性后续可再评估。
这个项目体现的产品判断是：AI 能力不是越自动越好，而是要在效率、可信度、可控性之间找到当前阶段最合适的边界。

Value

用户价值

用户不再需要记住“文件在哪”，只需要表达“我想要什么”，系统就能返回有依据的答案。

  • 信息获取方式从“翻目录和关键词搜索”变成“按语义和任务目标获取信息”。
  • 回答信任感从“模型说了什么”变成“模型说了什么，以及依据在哪里”。
  • 多轮体验从“每轮重复背景”变成“系统能接住上文继续做”。
  • 系统角色从“静态检索问答”升级为“动态决策与任务执行入口”。
  • 扩展方式从“继续堆功能”变成“通过 Tool 接入新能力”。
  • 风险控制从“能回答”推进到“可追溯、可解释、可控边界内执行”。

Limits & Next

限制与未来方向

这部分不是削弱项目，而是说明我知道系统边界在哪里，也知道下一步应该如何增强。

当前限制
  • 小样本验证规模有限，尚未覆盖复杂长链任务。
  • Router 在问题意图模糊、上下文不足时仍可能误判任务类型。
  • 多工具协同能力尚未引入完整 Planner、状态管理和失败回滚。
未来方向
  • 多工具 Agent：增加摘要、对比、索引更新和写回确认类工具。
  • Planner 任务拆解：把复杂目标拆成可执行步骤。
  • 多 Agent 协作：Research / Product / Content / Review 分工。

My Role

我的角色

  • 问题定义与产品设计：明确目标用户、使用场景、核心问题和 MVP 范围。
  • 系统架构拆解：完成从 V1 基础 RAG 到 V2 Agentic RAG 的演进设计。
  • Agent 机制设计：拆分 Router、Tool、Memory、Agent Core。
  • Prompt 设计：约束是否检索、如何基于来源回答。
  • Demo 验证与评估体系：用真实任务问题验证检索、生成和 Agent 判断。
  • 产品化表达与作品集封装：把技术实现转成公开评审可理解的产品案例。

Summary

一句话总结

这个项目最想展示的不是我会做一个 RAG，而是我能从真实问题出发，把它迭代成一个具备决策、工具调用和边界控制的 Agentic RAG 产品。
