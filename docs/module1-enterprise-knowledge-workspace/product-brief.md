# 产品简报

## 用户问题

用户的有效知识分散在笔记、索引、读书材料、项目资料、学习文档和原始文件中。Obsidian 适合人工组织和双链，但跨笔记、跨文档的语义检索能力有限。

同时，很多资料具有私人属性，不适合上传到外部服务。因此系统需要优先满足本地隐私和可控性。

## 目标用户

- 个人知识工作者。
- 需要维护作品集、项目资料和研究笔记的 AI 产品经理候选人。
- 重度 Obsidian 用户。
- 本地文档存放在 vault 外，需要统一检索。
- 使用低配置电脑，希望默认方案轻量可跑。

## 核心场景

| 场景 | 用户需求 | 产品响应 |
|---|---|---|
| 找回规则 | “我之前在哪里写过文件管理规则？” | 对 Obsidian 笔记做语义检索 |
| 恢复项目上下文 | “这个 RAG 项目之前做了哪些决策？” | 返回来源路径、章节和片段 |
| 搜索本地文档 | “哪份学习资料提到这个概念？” | 转换并索引本地文档 |
| 带来源问答 | “基于我的笔记总结一下这个流程。” | 先召回 chunks，再基于来源回答 |
| 笔记改动后刷新 | “我改了笔记，索引怎么更新？” | 按文件路径刷新旧 chunks |

## MVP 范围

- 索引选定的 Obsidian Markdown 文件夹。
- 索引选定的本地原始文档。
- 将 DOCX、PDF、PPTX、TXT、Markdown 转换为文本。
- 按 Markdown 结构和目标字符长度切块。
- 本地生成 embedding。
- 使用 Chroma 存储向量。
- 支持 CLI 和网页 GUI 检索。
- 通过本地小模型生成带来源的轻量回答。

## 关键产品决策

### 先检索，再回答

MVP 优先保证检索质量。如果召回片段不可靠，RAG 生成答案也不会可靠。

### 本地优先

embedding、Chroma、转换缓存和日志留在本机，降低隐私风险，避免依赖云端向量数据库。

### Obsidian 和原始文档分层

Obsidian 仍然是人工维护的笔记层，本地文档库仍然是原始文件层，RAG 是额外增加的检索层。

### 轻量模型优先

测试机器是 2020 款 Intel MacBook Pro，16GB 内存，因此默认设计避免大模型，采用小批量索引。

## 成功指标

- 搜索能在 top 3 结果中召回预期来源。
- 搜索结果包含清晰路径和片段。
- RAG 回答必须引用检索来源。
- 修改文件后重新索引不会留下重复旧 chunks。
- 换电脑时可以根据文档和配置模板重建系统。

## 取舍

| 选择 | 收益 | 成本 |
|---|---|---|
| 本地 embedding | 隐私和控制力更强 | 索引更慢 |
| 本地 Chroma | MVP 持久化简单 | 迁移时建议重建 |
| 小模型回答 | 低配置电脑可跑 | 推理能力有限 |
| 分批索引 | 降低卡顿风险 | 全量索引耗时更长 |
| MVP 不做 OCR | 降低复杂度 | 扫描版 PDF 覆盖不足 |
