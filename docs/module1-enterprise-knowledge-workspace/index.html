项目 01｜知识型 AI 应用

企业知识工作台

从可信上下文到 Agent 工作流的产品化演进

模型都这么强了，为什么企业还需要自己的知识系统？

因为企业不只需要“一个像样的回答”，还需要知道答案来自哪里、是否适用于当前场景、资料是否更新、权限是否可控，以及出错后能否定位、修正和回滚。

从“能回答”到“敢采用”，企业知识 AI 的关键不是更聪明，而是有证据、有边界、可追溯、可回滚。

在企业场景中，答案本身不是终点。业务负责人真正关心的是：这句话从哪里来，是否适用于当前情境，出了错能否定位和修正。

业务问题

业务问题定义

本项目先定义企业知识复用与决策准备场景，再判断 AI 应该进入哪些知识工作节点。

资料分散

资料不在同一处

项目资料、会议记录、复盘文档和规则说明散落在不同位置。

经验难复用

经验留在人脑里

历史项目经验依赖个人记忆，新人或跨部门接手成本高。

依据难追溯

结论缺少来源

结论被复用时，来源、依据和历史判断难以确认。

复盘难沉淀

项目结束即沉默

材料留在文档里，没有进入可持续检索和迭代机制。

用户角色

用户角色与核心任务

这不是一个给所有人闲聊的知识库，而是服务于高频知识工作的角色。用户表面上是在“搜索资料”，实际是在完成一段需要依据、判断和沉淀的知识工作。

项目负责人 / 产品经理

快速找到历史判断

查找历史方案、复盘资料和决策依据，用于新项目判断和方案准备。

新接手成员

快速理解项目背景

补齐项目上下文，了解历史取舍、关键资料和已有结论。

管理者 / 决策参与者

确认结论是否可信

查看结论依据、历史案例和风险点，减少只看摘要带来的误判。

运营 / PMO / 知识管理角色

沉淀组织知识

维护项目经验，把重复沟通转成可检索、可复盘、可迭代的知识资产。

找资料→ 找依据→ 整理结论→ 项目复盘→ 决策准备→ 形成参考建议→ 沉淀经验→ 连续追问

流程对比

AI 接入前后：从人工翻资料到可追溯知识工作流

同样是“找资料”，接入 AI 工作台前后最大的差异不是速度，而是结论、依据、人审和后续沉淀是否被组织进同一条任务链。

改造前：没有 AI 工作台时

产生资料 散落存储 人工搜索 打开多个文档 手动对比 整理结论 写汇报或复盘 经验再次散落
  • 依赖关键词和路径记忆
  • 找到文件后仍要人工翻阅
  • 结论和依据分离
  • 复盘难以沉淀为组织资产

改造后：有 AI 工作台后

提出知识任务 AI 判断任务类型 检索指定资料 返回引用来源 生成结构化草稿 人工确认 进入复盘 / 经验沉淀 / eval set
  • AI 负责召回、整理、溯源
  • 人负责判断、确认、取舍
  • badcase 进入迭代机制
  • 输出进入可维护知识工作流

产品责任单元

产品责任单元定义

本项目不是“做一个问答机器人”，而是设计一个面向企业知识复用与决策准备的 AI 工作台。它负责将分散资料转化为可检索、可追溯、可复盘、可沉淀、可迭代的知识工作流。

知识检索

找到相关资料和依据。

引用溯源

降低幻觉风险，支持人工核验。

知识任务型问答

从“找答案”升级为“完成知识整理任务”。

项目复盘

把历史资料转成结构化复盘。

Memory 机制

保留任务上下文和当前目标。

badcase / eval

支持质量回归和持续迭代。

版本管理

保证工作流可追踪、可回滚。

关键产品判断

关键产品判断：什么时候需要可信上下文层？

不是所有 AI 应用都需要 RAG。如果只是通用问答、低风险文本生成、一次性内容整理，可以直接调用大模型。但当场景涉及指定资料、引用依据、权限边界、持续更新和组织记忆时，就必须在基础模型和企业知识资产之间设计可信上下文层。

RAG 不是最终价值，可信上下文层才是产品价值。它位于基础模型与企业知识资产之间，用于承接检索、引用、权限、记忆、评估和版本管理，让知识回答从“能生成”进入“可核验、可维护”的产品状态。

直接调用大模型

可作为生成层

适合通用问答、写作润色；不适合作为企业指定资料和来源追溯的知识来源。

关键词搜索

保留为辅助能力

适合明确文件名、术语、标题；不擅长模糊意图和跨文档总结。

模型微调

暂不优先

适合稳定格式、分类、风格；不适合频繁更新的企业知识和引用溯源。

RAG / 可信上下文层

V1 主方案

适合指定资料、引用溯源、知识更新，是企业知识回答可信度的基础设施。

Agent 工作流

V2 升级方向

适合连续知识任务、路由、工具调用和人审确认，让 AI 进入真实任务流。

问题直接调用大模型可信上下文层
企业内部资料不保证覆盖最新内部资料接入指定资料范围
引用与依据难以稳定追溯可回到原文核验
知识更新依赖模型或手动上下文可随资料库更新
权限边界难以管理可按资料范围设计
组织记忆难以沉淀可结合复盘、badcase 和版本记录

产品演进

产品演进路径：从可信上下文到可维护 Agent 工作流

V1 / V2 / V3 的演进不是“技术越来越复杂”，而是产品化要求越来越高：先解决可信上下文，再解决连续知识任务，最后解决评估、回滚和长期维护。

Agent 工作流

Agent 工作流：从单轮问答到连续知识任务

Agent 的第一步不是执行，而是判断任务路径。如果 Router 判断错，后面的工具再强也没有意义。因此页面要展示路由判断、工具调用、来源片段、回答结果和 Human Review 状态。

Router

判断用户是在查资料、做复盘、生成总结，还是需要澄清。

Tool

将检索、总结、复盘等能力封装为可调用工具。

可信上下文层

提供可追溯的知识依据。

Memory

保留当前项目背景和最近上下文。

Agent Core

组织多步骤执行链路。

Human Review

在关键输出前保留人工确认。

badcase 复盘

将错误案例沉淀为下一轮迭代素材。

执行轨迹

执行过程不是黑盒：让 Agent 的判断路径可见

用户输入后，系统先判断是直接回答、检索资料还是需要澄清，再决定是否调用 search_docs，并把来源片段、回答结果和 Human Review 状态展示给用户。

用户输入→ Router 判断直接回答 / 检索 / 澄清→ Tool 调用 search_docs→ 返回来源片段→ 结合 Memory 生成回答→ 展示答案 + 来源 + 执行过程→ 人工确认关键输出

可维护机制

可维护机制：让 AI 工作流不是一次性 Demo

如果这个知识工作台不是演示一次，而是持续服务一个团队，就必须回答：哪些信息能进入记忆？提示词和工作流如何记录版本？badcase 如何沉淀？修复后如何回归测试？失败版本如何回滚？哪些输出必须人工确认？

任务与记忆

plan.md / memory_policy.md

记录任务拆解、记忆范围、上下文清理规则，避免 Memory 变成无限堆积的聊天历史。

版本与判断

prompt_version.md / workflow_changelog.md / decision_log.md

记录提示词、工作流和关键产品判断的变化，让改动有原因、有版本、有回看路径。

验证与回滚

badcase_log.md / eval_set.md / rollback_note.md

把错误案例、测试问题和失败版本纳入持续维护，避免同类问题在下一版重复出现。

人工确认

human_review_checkpoints.md

明确哪些输出必须人工确认，尤其是复盘结论、决策建议和对外引用内容。

产品原则

可追踪、可评估、可回滚

让版本、问题、修复和人工确认都能被团队持续追踪。

工程脚手架不是为了炫技，而是为了让 AI 工作流可追踪、可评估、可回滚。

没有版本、badcase、eval set 和回滚记录，Agent 工作流即使演示成功，也很难进入持续迭代。

验证与迭代闭环

验证与 badcase 闭环

这个项目不是一次性 Demo，而是把错误输出、路径误判和引用问题纳入回归测试与版本迭代。每个 badcase 都要回答：问题发生在哪一层，怎么修复，沉淀出什么原则，如何进入 eval set。

维度验证重点
检索层正确来源是否进入召回结果。
生成层回答是否基于来源，是否承认不确定。
Agent 层Router 是否选对路径，Memory 是否串扰。
治理层badcase 是否进入回归测试，版本是否可追踪。
Badcase 01

语义相近导致误召回

问题现象
用户询问某历史项目复盘时，系统召回了另一个语义相近但业务对象不同的项目资料。
发生层级
检索层。
修复动作
增加 project_name、source_type、path 等元数据；query rewrite 时补充项目对象；增加来源筛选。
沉淀原则
相似度高不等于业务相关。企业知识检索不能只看向量相似度，还要结合业务对象和元数据过滤。
进入 eval set
加入“相似项目区分”测试问题，后续版本必须回归。
Badcase 02

来源不足但强行回答

问题现象
来源片段不足以支持结论时，模型仍生成了完整、流畅但依据不足的回答。
发生层级
生成层。
修复动作
增加引用覆盖检查；当来源不足时输出“不确定”和缺失依据；把需要人工补充的资料显式列出。
沉淀原则
可信度优先于流畅度。宁可承认不确定，也不要编造。
进入 eval set
加入“低证据回答”测试问题，检查回答是否主动降级并提示人工确认。
Badcase 03

Router 误判

问题现象
用户明确询问指定资料，但 Router 误判为直接回答，导致系统没有调用 search_docs。
发生层级
Agent 控制层。
修复动作
补充 Router 判定规则；将“指定项目、指定文档、历史复盘”等触发词优先路由到检索；记录路由原因。
沉淀原则
Agent 的第一步是判断任务路径。Router 错了，工具再强也没有意义。
进入 eval set
加入“指定资料必须检索”测试问题，检查路由、工具调用和来源片段是否完整。
步骤 1

badcase

记录错误输出、误召回或路径误判。

步骤 2

定位

判断问题来自检索、生成、Router 还是 Memory。

步骤 3

修复

调整规则、提示词、工具路径或评估样本。

步骤 4

回归测试

进入 eval set，防止同类问题反复出现。

项目边界

项目边界与后续扩展

这个项目的价值，不在于把企业知识系统一次性做完整，而在于验证一条可扩展的产品化路径：先建立可信上下文层，再让 AI 进入连续知识任务，最后用人审、评估、badcase 和版本机制把它变成可维护的工作流。

当前边界

  • 当前阶段是产品级原型 / 可演示 MVP。
  • 不声称真实企业生产部署。
  • 不做完整权限、审计、多用户和系统集成。
  • 不自动写入、删除、发布。
  • 不替代人工判断。

项目 1 与项目 2 的边界

项目 1：企业知识工作台

核心对象
知识资产、历史经验、决策依据
回答的问题
企业过去积累的资料、经验和判断，如何被检索、引用、复盘和沉淀。

项目 2：AI 业务需求流转工作台

回答的问题
