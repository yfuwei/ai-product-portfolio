02 · Business Problem

业务问题定义

本项目先定义企业知识复用与决策准备场景，再判断 AI 应该进入哪些知识工作节点。

资料分散

资料不在同一处

项目资料、会议记录、复盘文档和规则说明散落在不同位置。

经验难复用

经验留在人脑里

历史项目经验依赖个人记忆，新人或跨部门接手成本高。

依据难追溯

结论缺少来源

结论被复用时，来源、依据和历史判断难以确认。

复盘难沉淀

项目结束即沉默

材料留在文档里，没有进入可持续检索和迭代机制。

03 · AI Intervention

AI 介入点设计

AI 不是只用于“文档问答”这一个点，而是嵌入知识工作流的多个节点：找资料、找依据、整理结论、项目复盘、经验沉淀和迭代优化。

工作节点AI 介入点产品价值
找资料语义检索 + 相关片段召回降低查找成本
找依据引用溯源 + 证据片段让结论可核验
整理结论多文档归纳 + 结构化总结提升整理效率
项目复盘复盘模板 + 关键经验提炼沉淀组织记忆
迭代优化badcase + eval + 版本记录支持持续改进

04 · Product Unit

产品责任单元定义

本项目不是“做一个问答机器人”，而是设计一个面向企业知识复用与决策准备的 AI 工作台模块。它负责将分散资料转化为可检索、可追溯、可复盘、可沉淀、可迭代的知识工作流。

知识检索

找到相关资料和依据。

引用溯源

降低幻觉风险，支持人工核验。

知识任务型问答

从“找答案”升级为“完成知识整理任务”。

项目复盘

把历史资料转成结构化复盘。

Memory 机制

保留任务上下文和当前目标。

badcase / eval

支持质量回归和持续迭代。

版本管理

保证工作流可追踪、可回滚。

05 · Product Judgment

基础模型增强后，为什么仍需要可信上下文层？

基础模型能力增强后，很多通用问答和低风险文本生成场景确实可以直接由模型完成。但在企业知识复用与决策准备场景中，核心问题不是模型能不能生成一个看似合理的答案，而是答案是否来自指定资料、是否能追溯依据、是否能随知识更新、是否符合权限边界、是否能被人工核验。

因此，本项目中 RAG 不被表达为最终产品主角，而是被重新定位为知识与上下文中间层：它位于基础模型与企业知识资产之间，用于承接检索、引用、权限、记忆、评估和版本管理。

问题直接调用基础模型可信上下文层
企业内部资料不保证覆盖最新内部资料接入指定资料范围
引用与依据难以稳定追溯可回到原文核验
知识更新依赖模型或手动上下文可随资料库更新
权限边界难以管理可按资料范围设计
组织记忆难以沉淀可结合复盘、badcase 和版本记录

06 · Evolution

从可信检索到 Agent 工作流，再到可维护 MVP

V1 / V2 / V3 的演进不是“技术架构越来越复杂”，而是企业知识工作流从可信上下文，到任务执行，再到可维护机制的产品化演进。

阶段核心问题产品判断
V1：可信知识检索为什么需要可信上下文层？先解决资料找得到、答案有依据、结论可复核。
V2：Agent 工作流为什么 RAG 不够？用户要完成连续知识任务，需要 Router、Tool、Memory 和 Human Review。
V3：可维护 MVP为什么 Agent 还不够？没有 eval、badcase、changelog、rollback，AI 工作流仍然只是一次性 demo。

07 · Agent Workflow

Agent 工作流：从单轮问答到连续知识任务

Router

判断用户是在查资料、做复盘、生成总结，还是需要澄清。

Tool

将检索、总结、复盘等能力封装为可调用工具。

可信上下文层

提供可追溯的知识依据。

Memory

保留当前项目背景和最近上下文。

Agent Core

组织多步骤执行链路。

Human Review

在关键输出前保留人工确认。

failure case / badcase 复盘

将错误案例沉淀为下一轮迭代素材。

08 · Evaluation Loop

验证与 badcase 闭环

这个项目不是一次性 demo，而是把错误输出、路径误判和引用问题纳入回归测试与版本迭代。

维度验证重点
检索层正确来源是否进入召回结果。
生成层回答是否基于来源，是否承认不确定。
Agent 层Router 是否选对路径，Memory 是否串扰。
治理层badcase 是否进入回归测试，版本是否可追踪。
badcase 01

语义相近导致误召回

说明相似度不等于业务相关性，需要结合问题类型、资料范围和证据片段判断。

badcase 02

来源不足但强行回答

要求系统承认不确定，并把缺失依据暴露给人工确认。

badcase 03

Router 误判

体现 Agent 控制层风险，需要把路径选择错误纳入回归样本。

Step 1

Badcase

记录错误输出、误召回或路径误判。

Step 2

定位

判断问题来自检索、生成、Router 还是 Memory。

Step 3

修复

调整规则、提示词、工具路径或评估样本。

Step 4

回归测试

进入 eval set，防止同类问题反复出现。

09 · Boundary

项目边界与后续扩展

当前边界

  • 当前是产品级原型 / 可演示 MVP，不是真实企业生产系统。
  • 不做完整权限系统，不做完整任务管理系统。
  • 不自动写入 / 删除 / 发布，不替代人工判断。
  • 企业化扩展包括权限、审计、多用户、系统连接和评估面板。

模块 1 与模块 2 的边界

模块核心对象回答的问题
模块 1：企业知识工作台 知识资产、历史经验、决策依据 企业过去积累的资料、经验和判断，如何被检索、引用、复盘和沉淀。
模块 2：AI 业务需求流转工作台 当前需求、问题、风险、责任链路、状态流转 新业务信号产生后，如何被识别、确认、分流、追踪和复盘。
